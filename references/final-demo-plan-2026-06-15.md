# 资源管理 Agent 最终演示 Demo 计划（2026-06-15 前）

## 目标

在 2026-06-15 前交付一个可以用于参赛/汇报的最终演示版本，重点证明：

- 这不是单点脚本集合，而是可交接、可配置、可治理的业务 Agent。
- VM 可用自然语言触发节点，Agent 从 Lark 读取真实状态并调用确定性脚本执行。
- 每一步有输入、输出、日志、人工确认、安全边界和回归验证。
- Demo 使用脱敏测试数据 / TEST_MODE / 草稿模式，但展示真实执行路径，不伪造核心输出。

## 最终产物

1. 演示视频：3-4 分钟，适合比赛和管理层观看。
2. 证据目录：保存真实运行 transcript、stdout/stderr、summary。
3. HTML/README 说明页：用于补充能力范围、治理机制、验证结果。
4. 录制脚本与素材：可重复生成，不依赖临时对话上下文。

## 视频结构

### 1. 开场：Agent 成熟度定位（约 20 秒）

展示 README / 项目标题 / 关键指标：

- 8 个 Skill 能力节点
- 16 个业务状态节点
- 6 个自动执行节点
- 4 个固定人工确认节点
- Lark 为事实数据库，脚本确定性执行，LLM 只处理必要智能节点

### 2. VM 自然语言触发（约 40 秒）

展示 VM 输入：

```text
调用资源管理 Agent，看下青木遥的简历
```

期望展示：

- 候选人定位：从 Lark 表读取青木遥 record
- 附件读取：下载/解析简历附件
- LLM 提取：结构化字段
- 规则评分：价格、资历、微调、总分、评级
- checkpoint：只展示 VM 需要确认的结论

关键输出应接近生产终端风格：

```text
青木遥 的评分结论：
- 总分：100/100
- 档位：S
- 价格：50/50
- 资历：50/50
- 微调：+2
- AI建议：优先录用
- 有效简历：是
- 依据：PDF 解析成功，识别到游戏本地化、翻译/项目经验、明确字数依据
```

### 3. 测试邀约节点（约 40 秒）

展示 VM 输入：

```text
给青木遥发送测试题，附件用 /path/to/test.xlsx
```

期望展示：

- 前置条件检查：候选人、邮箱、附件、当前状态
- 邮件草稿/TEST_MODE 发送：展示收件人安全边界
- Lark 写回字段：状态、测试发送时间、workflow_log
- checkpoint：VM 确认邮件内容或发送结果

### 4. 合同生成节点（约 50 秒）

展示 VM 输入：

```text
给青木遥准备合同
```

期望展示：

- 读取合同信息收集表
- 按账户类型、账户地区、币种选择合同模板
- 自动填充变量并输出“已填字段 / 缺失字段 / 待人工确认字段”
- 生成 docx 或草稿路径
- 高风险动作保留人工确认

### 5. 签字核查 / Badcase 回流 / 治理能力（约 50 秒）

展示三个成熟度点，不必完整跑很久：

- 签字合同核查：视觉模型/差异摘要/人工确认
- Badcase：VM 自然语言标记，Agent 自动生成脱敏 snapshot
- 全语种评分规则覆盖：22/22 主流市场语言对覆盖检查 PASS

### 6. 收束：生产级边界（约 20 秒）

强调：

- Lark 保存状态，不依赖 LLM 记忆流程
- 单点能力可独立调用，也可被 workflow 串联
- 关键节点人工确认
- 有 regression report / pricing coverage / privacy scan / integration readiness

## 验收标准

### 必须满足

- 视频中不出现真实 API key、邮箱密码、银行号、证件号、完整合同敏感正文。
- 核心输出来自真实脚本或真实 transcript，不使用纯手写假输出冒充运行结果。
- 评分、测试题、合同至少覆盖 3 个关键节点。
- 至少展示 1 个 checkpoint/人工确认节点。
- 至少展示 1 个治理能力：Badcase、trace/log、全语种覆盖、回归报告四选一，建议展示全语种覆盖。
- 最终 README 与视频能力描述一致。

### 不强求

- 不要求演示全自动从第 1 节点跑到第 16 节点。
- 不要求生产环境真实发送邮件或真实写合同敏感信息。
- 不要求前端成为最终主载体；前端可作为辅助画面。
